# AGENTS.md

本文件用于规范 Codex 后续开发本项目时的工作方式。

项目定位：Python 控制台修仙文字游戏。当前阶段优先打磨第一章炼气期体验，不做 GUI，不做第二章，不做大型宗门系统，不做引擎迁移。当前项目已推进到 v0.1.30 之后，后续开发应继续围绕第一章闭环、路线平衡、事件池、盗术、黑水、装备和大比体验展开。

## 开发边界

1. 不要随意重构整个项目结构。
2. 不要破坏旧存档兼容。
3. 不要新增 GUI、第二章、大型宗门系统或引擎迁移。
4. 不要为了局部功能引入过重抽象；优先沿用现有文件和数据结构。
5. 新玩法应先服务第一章炼气期闭环。

## 模拟与平衡规则

1. 新玩法系统必须接入 `simulate.py`。
2. 数值改动必须运行模拟并汇报结果。
3. 不要为了通过测试硬改评价阈值。
4. 不要让任何路线无成本碾压。
5. 不要把高风险路线调成稳定刷资源路线。
6. 不要只看单次模拟结论；重大数值改动必须关注 200 局和 1000 局稳定性。

## 提交前验证

提交前必须运行：

```powershell
python -m py_compile *.py
python simulate.py
python simulate.py --runs 5
```

重大数值改动额外运行：

```powershell
python simulate.py --runs 200
python simulate.py --runs 1000
```

如果本次只修改文档，可以不运行模拟，但最终汇报中必须明确说明未改运行逻辑。

提交规则：

- 除非用户明确要求先审查，否则完成任务并通过必要检查后，可以自动提交并推送。
- 不允许 `git push --force`。
- 不建议使用 `git add .`，除非已经确认工作区只有本次应该提交的文件。
- 每次提交前必须检查 `git status --short`。

## 旧存档兼容

涉及 `Player` 字段、存档结构、配置读取或版本迁移时，必须保证旧存档可读。

新增字段应满足：

- 有合理默认值。
- `from_dict()` 能自动补字段。
- `to_dict()` 能稳定写出字段。
- `clamp()` 和状态显示不会因旧存档缺字段崩溃。

## 后续开发重点

当前项目已推进到 v0.1.30 之后。后续开发应优先服务第一章炼气期闭环，不要回退到旧版本设计假设。

后续重点包括：

- 路线平衡，尤其是厚积薄发流、纯打坐流的稳定性压制。
- 第一章事件池继续扩展。
- 盗术高阶内容，包括偷气运、偷机缘、偷因果、偷寿元、偷 NPC 传承。
- 黑水坊市风险反馈与投机路线体验。
- 装备、事件、大比之间的轻量联动。
- 随心混合流的专属反馈和保底体验。

盗术内容设计要求：

- 基础盗窃
- 高阶盗术
- 偷气运
- 偷机缘
- 偷因果
- 偷寿元
- 偷 NPC 传承

1. 高阶盗术必须高风险、高门槛、低频率。
2. 盗术不能成为无成本刷资源路线。
3. 偷气运、偷机缘、偷因果、偷寿元、偷 NPC 传承不能做成稳定收益按钮。
4. 失败后果要可见，包括赔偿、拒赔、强行脱身、结仇、声望下降、暴露、业力、追踪标记等。
5. 未被发现的偷窃，不应立刻增加公开声望惩罚；可以增加隐性痕迹、因果、追踪、心理压力或嫌疑值。
6. 盗术投机流必须保留波动性，有爆点但不能稳定碾压其他路线。
7. 随心流可以低概率接触盗术，但不能被迫变成盗术流。

## 黑水坊市约束

1. 不要随意大改 `heishui_market.py` 主逻辑。
2. 不要为了修复模拟提示而强行扩展黑水内容。
3. 黑水投机流应保持高风险高收益，不应被削成不可玩，也不应成为稳定最优解。
